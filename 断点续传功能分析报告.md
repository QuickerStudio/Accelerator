### 2. 回答你的问题

#### Q1: 断点续传是接着继续下载还是重新下载？
**答案：接着继续下载**

- 每次点击继续下载时，会检查文件是否存在
- 如果文件存在且大小 > 0，会发送 HTTP Range 请求：`Range: bytes=<已下载大小>-`
- 服务器返回 206 Partial Content 时，只传输剩余部分
- 使用追加模式写入：`FileOutputStream(targetFile, append = true)`

#### Q2: 是否会清空缓存？
**答案：不会清空缓存**

- 已下载的数据会保留在文件中
- 只有在取消下载时才会删除文件（第 111-113 行）
- 暂停下载时文件完整保留

#### Q3: 是否会重置目录中的数据？
**答案：不会重置**

- 文件保持在原位置：`context.filesDir/models/gemma-3n-e2b-it-int4.litertlm`
- 目录结构不变
- 只有文件内容会追加新数据

#### Q4: 从连接中获取数据的模式是什么？
**答案：HTTP Range 请求 + 追加写入模式**

1. **请求模式：**
   - 发送 `Range: bytes=<existingSize>-` 请求头
   - 服务器返回 206 Partial Content（部分内容）
   - 只传输从断点位置到文件末尾的数据

2. **写入模式：**
   - 使用 `FileOutputStream(targetFile, append = true)` 追加模式
   - 新数据追加到已有数据后面
   - 不覆盖已下载的部分

## 二、发现的问题

### 🔴 严重问题 1：服务器不支持 Range 时会导致文件损坏

**问题描述：**
- 当前代码假设服务器总是支持 Range 请求
- 如果服务器不支持，会返回 200 OK 而不是 206 Partial Content
- 但代码仍然使用追加模式写入，导致：**旧数据 + 完整新数据 = 文件损坏**

**示例场景：**
```
1. 用户下载了 1GB 数据后暂停
2. 继续下载时，服务器不支持 Range，返回 200 OK
3. 服务器发送完整的 3.6GB 数据
4. 代码以追加模式写入
5. 最终文件大小：1GB + 3.6GB = 4.6GB（损坏）
```

**当前代码逻辑：**
```kotlin
val append = responseCode == HttpURLConnection.HTTP_PARTIAL  // 只检查 206
java.io.FileOutputStream(targetFile, append).use { output ->
    // 如果 responseCode == 200，append 仍为 false，但...
    // 问题：totalSize 计算错误，会导致进度显示异常
}
```

**实际问题：**
```kotlin
val totalSize = if (responseCode == HttpURLConnection.HTTP_PARTIAL) {
    existingSize + connection.contentLength.toLong()  // 206: 正确
} else {
    connection.contentLength.toLong()  // 200: 应该删除旧文件！
}
```

### 🔴 严重问题 2：切换下载线路时没有清理旧文件

**问题描述：**
- 用户从 HuggingFace 下载了 1GB 后暂停
- 切换到 ModelScope 线路继续下载
- 从不同源续传可能导致文件损坏（不同服务器的文件可能有细微差异）

**当前代码：**

### 🟡 中等问题 3：没有验证 Content-Range 响应头

**问题描述：**
- 即使服务器返回 206，也应该验证 Content-Range 是否匹配请求
- 例如：请求 `Range: bytes=1000-`，应该验证响应 `Content-Range: bytes 1000-3655827455/3655827456`

**缺失的验证：**

### 🟡 中等问题 4：没有文件完整性校验

**问题描述：**
- 下载完成后只检查文件大小，没有验证文件哈希
- 网络传输可能导致数据损坏
- 应该添加 MD5 或 SHA256 校验

**当前验证：**

**问题描述：**

## 三、修复建议

### 修复 1：正确处理服务器不支持 Range 的情况

```kotlin
// DEngine.kt 修改建议
suspend fun download(
    url: String,
    targetFile: File,
    onProgress: (downloaded: Long, total: Long, speed: Long) -> Unit
): Result<File> = withContext(Dispatchers.IO) {
    try {
        isPaused = false
        isCancelled = false
        targetFile.parentFile?.mkdirs()

        val connection = URL(url).openConnection() as HttpURLConnection
        connection.connectTimeout = 30000
        connection.readTimeout = 30000

        // 检查已下载的部分
        val existingSize = if (targetFile.exists()) targetFile.length() else 0L
        if (existingSize > 0) {
            connection.setRequestProperty("Range", "bytes=$existingSize-")
        }

        connection.connect()
        val responseCode = connection.responseCode

        // 🔧 修复：正确处理不同的响应码
        val (totalSize, append) = when (responseCode) {
            HttpURLConnection.HTTP_PARTIAL -> {
                // 206: 服务器支持断点续传
                val contentRange = connection.getHeaderField("Content-Range")
                // 可选：验证 Content-Range 是否匹配
                Pair(existingSize + connection.contentLength.toLong(), true)
            }
            HttpURLConnection.HTTP_OK -> {
                // 200: 服务器不支持 Range，需要重新下载
                if (existingSize > 0) {
                    // 删除旧文件，重新开始
                    targetFile.delete()
                }
                Pair(connection.contentLength.toLong(), false)
            }
            else -> {
                throw Exception("Unexpected response code: $responseCode")
            }
        }

        connection.inputStream.use { input ->
            java.io.FileOutputStream(targetFile, append).use { output ->
                val buffer = ByteArray(8192)
                var downloaded = if (append) existingSize else 0L
                var bytes: Int
                var lastUpdateTime = System.currentTimeMillis()
                var lastDownloaded = downloaded
                var speed = 0L

                onProgress(downloaded, totalSize, speed)

                while (input.read(buffer).also { bytes = it } != -1) {
                    if (isCancelled) {
                        throw Exception("下载已取消")
                    }

                    while (isPaused && !isCancelled) {
                        Thread.sleep(100)
                    }

                    output.write(buffer, 0, bytes)
                    downloaded += bytes

                    val currentTime = System.currentTimeMillis()
                    val timeDiff = currentTime - lastUpdateTime
                    if (timeDiff >= 1000) {
                        speed = ((downloaded - lastDownloaded) * 1000) / timeDiff
                        lastUpdateTime = currentTime
                        lastDownloaded = downloaded
                    }

                    onProgress(downloaded, totalSize, speed)
                }

                onProgress(totalSize, totalSize, 0L)
            }
        }

        Result.success(targetFile)
    } catch (e: Exception) {
        if (isCancelled && targetFile.exists()) {
            targetFile.delete()
        }
        Result.failure(e)
    }
}
```

```kotlin
    )

    return true
}
```

```kotlin
fun validateFileHash(): Boolean {
    if (!modelFile.exists()) return false

    val expectedMd5 = configManager.getModelMd5() ?: return true  // 如果没有配置 MD5，跳过
    val actualMd5 = calculateMd5(modelFile)

    return expectedMd5.equals(actualMd5, ignoreCase = true)
}

private fun calculateMd5(file: File): String {
    val md = MessageDigest.getInstance("MD5")
    file.inputStream().use { input ->
        val buffer = ByteArray(8192)
        var bytes: Int
        while (input.read(buffer).also { bytes = it } != -1) {
            md.update(buffer, 0, bytes)
        }
    }
    return md.digest().joinToString("") { "%02x".format(it) }
}
```

## 五、总结

### 当前行为
✅ **正确的部分：**
- 不会清空缓存，保留已下载数据
- 不会重置目录
- 使用 HTTP Range 请求接着继续下载
- 追加模式写入新数据

❌ **存在的问题：**
1. 服务器不支持 Range 时会导致文件损坏（严重）
2. 切换线路时没有清理旧文件（严重）
3. 没有验证 Content-Range 响应头（中等）
4. 没有文件完整性校验（中等）
5. 进度计算在某些情况下不准确（轻微）

### 建议优先级
1. **高优先级：** 修复问题 1 和 2（防止文件损坏）
2. **中优先级：** 添加文件哈希校验
3. **低优先级：** 优化进度显示和错误提示
