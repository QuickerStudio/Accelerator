# 断点续传功能修复说明

## 修复的问题

### 🔴 严重问题：服务器不支持 Range 请求时导致文件损坏

**问题描述：**
- 当服务器不支持断点续传时，会返回 200 OK 而不是 206 Partial Content
- 旧代码仍然使用追加模式写入，导致：旧数据 + 完整新数据 = 文件损坏

**修复方案：**
1. 在 `DEngine.kt` 中添加响应码检查
2. 当收到 200 响应时，删除旧文件重新下载
3. 当收到 206 响应时，使用追加模式继续下载
4. 添加回调函数通知 Range 支持状态

---

## 修改的文件

### 1. DEngine.kt
**修改内容：**
- 添加 `onRangeSupportDetected` 回调参数
- 添加响应码检查逻辑（200 vs 206）
- 当服务器返回 200 时，删除旧文件重新下载
- 当服务器返回 206 时，使用追加模式继续下载

**关键代码：**
```kotlin
when (responseCode) {
    HttpURLConnection.HTTP_PARTIAL -> {
        // 206: 服务器支持断点续传
        supportsRange = true
        totalSize = existingSize + connection.contentLength.toLong()
        append = true
        startOffset = existingSize
    }
    HttpURLConnection.HTTP_OK -> {
        // 200: 服务器不支持断点续传
        if (existingSize > 0) {
            supportsRange = false
            targetFile.delete()  // 删除旧文件
        }
        totalSize = connection.contentLength.toLong()
        append = false
        startOffset = 0L
    }
}
```

### 2. DConfig.kt
**修改内容：**
- 在默认配置中为每个下载线路添加 `supportsRange` 和 `rangeChecked` 字段
- 添加 `updateRouteRangeSupport()` 方法保存 Range 支持状态
- 添加 `isRangeSupportedByUrl()` 方法查询 Range 支持状态

**新增方法：**
```kotlin
fun updateRouteRangeSupport(url: String, supportsRange: Boolean)
fun isRangeSupportedByUrl(url: String): Boolean?
```

### 3. DState.kt
**修改内容：**
- 更新 `DRoute` 数据类，添加 `supportsRange` 和 `rangeChecked` 字段

**修改后的数据类：**
```kotlin
data class DRoute(
    val name: String,
    val displayName: String,
    val url: String,
    val supportsRange: Boolean = true,
    val rangeChecked: Boolean = false
)
```

### 4. DManager.kt
**修改内容：**
- 更新 `downloadModel()` 方法，传递 `onRangeSupportDetected` 回调
- 在回调中保存 Range 支持状态到配置文件
- 修改 `switchRoute()` 方法，切换线路时删除不完整的文件

**关键修改：**
```kotlin
// 下载时检测并保存 Range 支持状态
val result = downloadEngine.download(
    url = currentUrl,
    targetFile = modelFile,
    onProgress = onProgress,
    onRangeSupportDetected = { url, supportsRange ->
        configManager.updateRouteRangeSupport(url, supportsRange)
        if (!supportsRange) {
            configManager.addDownloadLog("⚠️ 服务器不支持断点续传，已切换为完整下载模式")
        }
    }
)

// 切换线路时删除不完整的文件
fun switchRoute(): Boolean {
    if (stateMonitor.fileExists() && !stateMonitor.isFileComplete()) {
        stateMonitor.deleteFile()
        configManager.clearDownloadState()
    }
    // ... 切换逻辑
}
```

## 工作流程

### 场景 1：服务器支持断点续传（正常情况）
1. 用户下载 1GB 后暂停
2. 继续下载时，发送 `Range: bytes=1073741824-` 请求
3. 服务器返回 206 Partial Content
4. 使用追加模式写入剩余数据
5. 保存 `supportsRange: true` 到配置

### 场景 2：服务器不支持断点续传
1. 用户下载 1GB 后暂停
2. 继续下载时，发送 `Range: bytes=1073741824-` 请求
3. 服务器返回 200 OK（忽略 Range 请求）
4. **删除旧文件**，重新开始下载
5. 使用覆盖模式写入完整数据
6. 保存 `supportsRange: false` 到配置

### 场景 3：切换下载线路
1. 用户从 HuggingFace 下载 1GB 后暂停
2. 切换到 ModelScope 线路
3. **自动删除不完整的文件**
4. 清除下载状态
5. 从新线路重新开始下载

## 总结

✅ **已修复的问题：**
1. 服务器不支持 Range 时会正确删除旧文件重新下载
2. 切换线路时会自动清理不完整的文件
3. Range 支持状态会保存到配置文件中
4. 下载模式会根据服务器响应自动调整

✅ **防止的问题：**
1. 文件损坏（旧数据 + 新数据混合）
2. 从不同源续传导致的文件不一致
3. 进度显示错误

✅ **改进的功能：**
1. 自动检测服务器 Range 支持
2. 智能选择下载模式（追加 vs 覆盖）
3. 更详细的日志记录
4. 更安全的线路切换
