## 模块职责分析

### DManager (核心协调者)
**职责：**
- 编排整个下载流程
- 调用 DPing 检查网络
- 调用 DState 监控文件状态
- 调用 DEngine 执行下载
- 调用 DConfig 持久化配置
- 线路切换决策

**依赖关系：**
```kotlin
DManager {
    - configManager: DConfig      // 配置管理
    - stateMonitor: DStateMonitor // 状态监控
    - networkPing: DPing          // 网络检查
    - downloadEngine: DEngine     // 下载引擎
}
```

### DPing (网络检查器)
**职责：**
- 检查网络连接
- 检测服务器 Range 支持
- 生成下载配置建议

**特点：**
- 独立模块，无依赖
- 只负责检测，不修改状态

### DState (状态监控器)
**职责：**
- 监控文件系统状态
- 检查文件完整性
- 提供文件信息

**依赖关系：**
```kotlin
DStateMonitor {
    - configManager: DConfig  // 读取配置
    - modelFile: File         // 直接访问文件系统
}
```

**特点：**
- 直接与文件系统交互
- 通过 DConfig 读取配置
- 不修改配置，只读取

### DEngine (下载引擎)
**职责：**
- 执行实际的 HTTP 下载
- 处理断点续传逻辑
- 管理暂停/恢复/取消

**特点：**
- 独立模块，无依赖
- 只负责下载，不管理状态

### DConfig (配置管理器)
**职责：**
- 读写 config.json
- 持久化下载状态
- 管理日志记录

**特点：**
- 单一真相来源
- 所有配置的持久化都通过它

## 正确的数据流

### 场景1：开始下载

```
用户点击下载
    ↓
DManager.downloadModel()
    ↓
1. DState.isFileComplete() ← 检查文件是否已完整
    ↓
2. DPing.checkServer(url) ← 检查网络和 Range 支持
    ↓
3. DConfig.updateRouteRangeSupport() ← 保存检测结果
    ↓
4. DState.getFileSize() ← 获取已下载大小
    ↓
5. DConfig.updateDownloadState() ← 更新下载状态
    ↓
6. DEngine.download() ← 执行下载
    ↓
7. DState.validateFile() ← 验证文件完整性
    ↓
8. DConfig.updateDownloadState() ← 保存最终状态
```

### 场景2：切换线路

```
用户点击切换线路
    ↓
DManager.switchRoute()
    ↓
1. DState.fileExists() ← 检查文件是否存在
    ↓
2. DState.isFileComplete() ← 检查是否完整
    ↓
3. DState.deleteFile() ← 删除不完整的文件
    ↓
4. DConfig.clearDownloadState() ← 清除下载状态
    ↓
5. DConfig.addDownloadLog() ← 记录日志
```

## 问题分析

### ❌ 当前问题：DManager 和 DState 的关系混乱

**问题1：DState 依赖 DConfig**
```kotlin
// DStateMonitor.kt
class DStateMonitor(private val context: Context) {
    private val configManager = DConfig(context)  // ❌ 创建了自己的 DConfig 实例
}
```

这导致：
- DManager 有一个 DConfig 实例
- DStateMonitor 又创建了另一个 DConfig 实例
- 两个实例可能不同步

**问题2：职责不清晰**
- DState 应该只负责监控文件系统状态
- 但它现在还要读取配置（expectedSize, sizeTolerance）
- 这些配置应该由 DManager 传递给它

## 正确的架构应该是

### 方案1：DState 完全独立（推荐）

```kotlin
// DStateMonitor 不依赖 DConfig
class DStateMonitor(private val context: Context) {
    private val modelFile: File
        get() = File(context.filesDir, "models/gemma-3n-e2b-it-int4.litertlm")

    // 所有需要配置的方法都通过参数传递
    fun isFileComplete(expectedSize: Long, sizeTolerance: Long): Boolean {
        val fileSize = modelFile.length()
        val sizeDiff = abs(fileSize - expectedSize)
        return sizeDiff <= sizeTolerance
    }

    fun validateFile(expectedSize: Long, sizeTolerance: Long): FileValidationResult {
        // ...
    }
}

// DManager 负责传递配置
class DManager(private val context: Context) {
    private val configManager = DConfig(context)
    private val stateMonitor = DStateMonitor(context)

    fun isModelComplete(): Boolean {
        val expectedSize = configManager.getExpectedModelSize()
        val sizeTolerance = configManager.getSizeTolerance()
        return stateMonitor.isFileComplete(expectedSize, sizeTolerance)
    }
}
```

### 方案2：DState 通过 DManager 访问配置

```kotlin
// DStateMonitor 接收配置对象
class DStateMonitor(
    private val context: Context,
    private val config: DConfig  // 接收 DManager 的配置实例
) {
    // 使用传入的配置实例
    fun isFileComplete(): Boolean {
        val expectedSize = config.getExpectedModelSize()
        val sizeTolerance = config.getSizeTolerance()
        // ...
    }
}

// DManager 传递配置
class DManager(private val context: Context) {
    private val configManager = DConfig(context)
    private val stateMonitor = DStateMonitor(context, configManager)  // 传递配置
}
```

## 推荐的最终架构

```
┌─────────────────────────────────────────────────────────────┐
│                        DManager                              │
│  (唯一的协调者，持有所有配置和状态)                            │
└──────┬──────────┬──────────┬──────────┬────────────────────┘
       │          │          │          │
       │          │          │          │
       ▼          ▼          ▼          ▼
   ┌──────┐  ┌──────┐  ┌──────┐  ┌──────────┐
   │DPing │  │DState│  │DEngine│  │DConfig   │
   │      │  │      │  │       │  │          │
   │独立   │  │独立   │  │独立    │  │单例      │
   └──────┘  └──────┘  └───────┘  └────┬─────┘
                                       │
                                       ▼
                                 ┌──────────┐
                                 │config.json│
                                 └──────────┘
```

**核心原则：**
1. DManager 是唯一的协调者
2. DConfig 是唯一的配置管理者
3. DState、DPing、DEngine 都是无状态的工具类
4. 所有配置都通过 DManager 传递
5. 避免模块之间直接依赖

你觉得应该用哪个方案？我倾向于方案2，因为它改动最小。
